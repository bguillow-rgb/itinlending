# SEO & AEO Plan

This project follows the global SEO/AEO/content playbook in `~/.claude/CLAUDE.md`.
This doc records how that playbook is implemented for the Itin sites and what's
planned. Goal: rank for ITIN long-tail queries **and** be the source AI answer
engines (AI Overviews, ChatGPT, Perplexity, Gemini, Copilot) cite.

## Content topology — hub-and-spoke

Three-tier architecture, defined in `consts.ts` (`PILLAR`, `PRODUCTS`) and wired by
`RelatedLinks`:

| Tier | Pages | Role |
|---|---|---|
| **Pillar** | `/itin-loans` | Canonical authority; links down to every cluster. |
| **Cluster** | `/itin-mortgage`, `/itin-auto-loan`, `/itin-credit-cards`, `/itin-personal-loans`, `/itin-business-loans`, `/how-to-get-an-itin`, `/itin-vs-ssn` | Mid-funnel commercial intent; link up to pillar, sideways to siblings, down to detail articles. |
| **Detail** | `src/content/articles/*.md` (e.g. `itin-car-loan`, `itin-loan-with-bad-credit`) | Long-tail research questions; link up to the relevant cluster. |

`RelatedLinks` (`currentPath`) renders the internal-link mesh on every money page
and article. Hub-and-spoke linking is the documented lever for AI-citation lift.

## On-page structure (every article)

Enforced by the article model + the daily generator prompt:
- **Quick Answer** block (40–60 words) at top, marked **Speakable** (`#quick-answer`).
- **Question-format H2s**, each a self-contained ~134–167-word answer unit.
- A cited stat/number roughly every 150–200 words, attributed in prose.
- At least one **comparison table**.
- **FAQ** section (5+ Q&As) → FAQPage schema.
- Internal links to pillar/cluster pages.
- Original wording; first-hand tone.

## Structured data (JSON-LD)

Implemented as components in `src/components/schema/`, injected via the layout
`head` slot:

| Page type | Schema shipped |
|---|---|
| Homepage | Organization, WebSite |
| Money pages | Organization, BreadcrumbList, FAQPage, **Service** (FinancialService) |
| Articles | Article (+ Speakable on Quick Answer), FAQPage, BreadcrumbList |
| /articles index (EN + ES) | **CollectionPage + ItemList** |
| /about | Organization, AboutPage, Person/editorial entity |

`ServiceSchema.astro` emits a `FinancialService` (areaServed US, audience "ITIN
holders and foreign nationals", provider = the Organization, `inLanguage` from the
page locale) on every `MoneyPageLayout` page. `CollectionPageSchema.astro` emits a
`CollectionPage` whose `mainEntity` is an `ItemList` of the listed articles, on
both `/articles` and `/es/articles`. Both derive `inLanguage` via `localeFor`
(EN=`en-US`, ES=`es-419`).

Publisher entity = **Timberline Ventures LLC**; byline = **ITIN Lending Editorial
Team** (a legitimate E-E-A-T anchor where there's no single named author).
**Validation plan:** run key URLs through Google's Rich Results Test and keep GSC
Enhancements at 0 errors. The health monitor (`scripts/monitor.mjs`) also asserts
JSON-LD parses on `/`, `/about`, `/articles` on every run.

## AEO infrastructure

- **`robots.txt`** (`web/public/robots.txt`) — default allow-all **plus** an
  explicit allow-list of AI crawlers (GPTBot, OAI-SearchBot, ChatGPT-User,
  ClaudeBot, Claude-Web, anthropic-ai, PerplexityBot, Perplexity-User,
  Google-Extended, Applebot(-Extended), CCBot, Bytespider, Meta-ExternalAgent,
  cohere-ai) so a CDN default-deny never blocks them by accident. Declares the
  sitemap.
- **`llms.txt`** (`web/public/llms.txt`) — markdown describing the site, what an
  ITIN borrower can do, the core guides (with URLs), sections, the publisher
  entity, and citation notes. Cheap scaffolding (low proven lift) — content +
  schema do the real work.
- **Sitemap** — auto-generated by `@astrojs/sitemap` each build; `404` and
  `thank-you` filtered out. `sitemap-index.xml` is the submitted entry point.
- **hreflang + canonical** — emitted per page for EN↔ES counterparts (via
  `altPath`); monitor warns if missing. Belt-and-suspenders: the sitemap also
  carries reciprocal `<xhtml:link rel="alternate" hreflang=…>` (en/es/x-default)
  per URL via the `serialize()` hook in `astro.config.mjs` — Astro's built-in
  `i18n` sitemap option can't be used because EN is un-prefixed (`/foo`) while ES
  is path-prefixed (`/es/foo`).
- **schema `inLanguage` follows the page's language** — every schema component
  (`WebSiteSchema`, `AboutPageSchema`, `ArticleSchema`) derives the locale from
  the URL via `getLangFromUrl` + `localeFor` (in `i18n/ui.ts`): EN pages emit
  `en-US`, `/es` pages emit `es-419` (Latin-American Spanish). Previously hardcoded
  to the site's EN locale, which mislabeled every Spanish page as English. This is
  the single source of truth — don't reintroduce `SITE.locale` in schema.

## Weekly GSC diff report — `web/scripts/gsc-report.mjs`

Stateless **last-7-days-vs-prior-7-days** comparison via the Search Console API,
split **EN vs ES** by page path (`/es/`). For each locale it reports totals
(clicks/impressions/CTR/position) plus top impression gains/losses, CTR drops, and
near-page-1 queries. Implementation notes:
- No `googleapis` dependency — the RS256 JWT is minted directly with `node:crypto`.
- **Env-gated**: no-ops unless `GSC_SA_KEY` (service-account JSON) and
  `GSC_PROPERTY` are set, consistent with the `consts.ts` monetization gating.
- Driven by `.github/workflows/gsc-report.yml` (cron `0 14 * * 1`, Mondays, +
  `workflow_dispatch`). Until the GSC secret is wired, the workflow runs and cleanly
  no-ops. This is the start of the weekly technical-SEO audit loop.

## Spanish-locale (`/es`) blackout — diagnosis 2026-06-13 (rank action ③)

**Symptom:** the 2026-06-13 rank report shows the `/es` locale earning ~0 search
impressions across all three sites — **2 Spanish query rows total** (both 1
impression, position 80–98) vs **117 English rows**.

**On-site/technical layer audited — all correct, nothing to fix:**
- **hreflang** — `BaseLayout.astro` emits reciprocal `en` / `es` / `x-default`
  alternates derived from the actual URL via `altPath()`. Reciprocity is sound.
- **canonical** — each `/es` page self-canonicals to its own `/es/...` URL.
- **schema `inLanguage`** — every schema component (`ServiceSchema`,
  `ArticleSchema`, `CollectionPageSchema`, `AboutPageSchema`, `WebSiteSchema`)
  derives the locale with `localeFor(getLangFromUrl(...))` → `es-419` on `/es`
  pages. Not hardcoded to the default locale.
- **translation quality** — `/es` money-page **bodies** are genuinely, natively
  translated (full Spanish prose, not machine-stub or translated-chrome-only).
- **discovery** — `/es` URLs are in the sitemap (as `<loc>` and as reciprocal
  hreflang refs) and build cleanly.

**Conclusion:** this is **not** an on-page i18n defect. The cause is
indexation + authority + demand, not markup. Two live possibilities remain,
distinguished only by data the automated audit can't reach:
1. `/es` pages crawled-but-not-yet-indexed (new, low crawl priority) → fix by
   requesting indexing + IndexNow/Bing submission.
2. `/es` pages indexed but the domain lacks the authority to surface for
   competitive Spanish ITIN queries → 3–6-month authority play, nothing to "fix."

**Needs the user's browser (next step):** run GSC **URL Inspection** on 2–3 `/es`
money pages (e.g. `/es/itin-mortgage`, `/es/itin-credit-cards`) on each property
to read the coverage state — *Indexed* vs *Crawled – currently not indexed* vs
*Discovered – currently not indexed*. That single check tells us whether to push
for indexing (case 1) or invest in Spanish authority/content (case 2).

## Credit-score cluster: de-cannibalization + Speakable (2026-06-13)

**Cannibalization fixed.** On itincreditscore.com the money page
`/check-credit-score-with-itin` and the article
`/how-to-check-credit-score-with-itin-number` shared the *identical* title and
competed for the same query (article at ~pos 72, 216 impr). Resolved by
**retargeting the article** (EN + es-419) to a distinct intent — "get your free
credit **report**, all 3 bureaus" — leaving the money page to own "check your
**score**." Title/description/targetQuery/quickAnswer/intro/lead-FAQ changed; the
**slug was kept** to preserve the indexed page's equity, and the intro now hands
"score" intent up to the money page. This is the model for any future
two-pages-one-query collision: pick the canonical owner, retarget the other to a
genuinely distinct sub-intent, keep the slug, link up.

**Speakable on money pages.** `ArticleSchema.astro` already emits
`SpeakableSpecification` on `#quick-answer`. Money pages did not — `ServiceSchema`
is `FinancialService`, which can't legitimately host `speakable`. Fixed by adding a
separate **WebPage+Speakable** JSON-LD node in `MoneyPageLayout.astro` targeting
`#quick-answer`. Done on itincreditscore.com; **not yet propagated** to `~/Itin` and
`~/ITINCreditCard` MoneyPageLayouts (same pattern — quick follow-up).
